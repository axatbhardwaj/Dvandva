# Case Study: PR 353

Source PR: https://github.com/defi-com/monorepo/pull/353

Export date: 2026-05-12.

## Summary

PR 353 was a large documentation coverage and accuracy campaign in `defi-com/monorepo`.

The run used two agents:

- Claude as driver for iterations 1-85.
- Codex as reviewer for iterations 1-85.
- Codex as driver for iterations 86-118.
- Claude as reviewer for iterations 86-118.

The PR finished the D1 and D2 dimensions:

- D1: docs exist at expected paths.
- D2: docs are accurate against current code, infra, and git history.

D3 and D4 were deferred to a follow-up ticket because they would require public-item inline docs and worked examples across many library surfaces.

## What Worked

The driver/reviewer split worked.

Codex caught real inaccuracies while reviewing Claude-driven work:

- Wrong consumer counts.
- Public export inventories that missed symbols.
- Markdown tables broken by unescaped union pipes.
- README examples that no longer matched current types.
- Stale audit rows after the README was fixed.
- Historical planning docs presented as current state.

Claude caught and verified real issues while reviewing Codex-driven work:

- Infra docs that needed source-backed precision.
- Integration docs that needed current route and deployment checks.
- Historical docs that needed archival framing.
- Production-relevant findings discovered during docs verification.

The explicit role model helped because one agent had permission to challenge the other, not just continue the task.

## What Hurt

The coordination channel was too heavy:

- PR comments became both agent mailbox and archival record.
- Every handoff needed GitHub reads.
- Both agents posted under the same GitHub user, so attribution relied on text prefixes.
- Timer/polling cadence created idle turns.
- 116 commits in one PR made the review surface large.

## Main Lesson

The useful primitive was not "two agents talking in PR comments."

The useful primitive was:

```text
driver checkpoint -> adversarial review -> narrow fix or handback -> verified next checkpoint
```

Dvandva keeps that primitive and moves the high-frequency coordination to local baton files.

## Design Consequences

For future workflows:

- Use local baton files for agent handoffs.
- Use PR comments only for human-facing summaries.
- Run one autonomous goal at a time by default.
- Split broad campaigns into area PRs when possible.
- Make exit criteria machine-checkable before starting.
- Keep explicit role ownership: Claude does, Codex reviews/fixes.
