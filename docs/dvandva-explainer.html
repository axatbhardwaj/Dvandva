द्वन्द्व · DVANDVA — a compound of two equal, opposing members

vadi × prativadi

Two rival AI agents from different vendors build your software together — one proposes, one prosecutes. They never talk. They pass a baton file through a state machine that refuses to finish until the work is proven.

VADI · proposes, implements PRATIVADI · reviews, attacks
no daemon — a JSON file is the whole channel crash-tolerant: kill either side, rejoin at preflight engine-agnostic: Claude Code × Codex is the canonical pair walkaway or supervised

why it exists

Different models have different blind spots. Dvandva weaponizes that.

Decorrelated review

The reviewer is never the engine that produced the work. A cross-vendor pair catches what a same-model self-review structurally cannot see.

Proof, not prose

A run cannot end because an agent says it's done. Termination demands both role-bound approvals, artifacts that exist on disk, and a verification matrix fresher than the last code change.

Bounded arguments

Every review loop is capped. When the pair genuinely disagrees, the protocol escalates to you — it never spins.

the turn loop

One baton, one writer, one gate

Whoever the baton assigns acts; the other blocks on dvandva wait. A turn ends by proposing the entire next baton — and the engine validates it against the state machine before installing it. Wrong owner, stale checkpoint, missing evidence, dropped peer work, illegal edge: rejected with a machine-readable reason.

step zero — before any route

Every line shares the same first four stations

Whatever mode or profile a run picks, it cannot reach research_drafting without first passing through a mandatory question gate: vadi asks, the human answers, then prativadi asks follow-ups informed by those answers, and the human answers again. The write gate requires at least five questions combined, with both roles represented — a solo role cannot satisfy the floor alone.

step one

Pick a mode: what kind of work is this?

development

Ship a change. The full lifecycle — research, locked plan, phased implementation, adversarial review, evidence-gated termination.

profile chooses the ceremony ↓

research

Answer a question with reviewed evidence. Optionally ends by seeding a development plan for a follow-on run.

review

A verdict on an existing diff or artifact. Scope → deep review → fixes → verdict. Nothing gets built.

step two — the centerpiece

Pick a profile: every path, one map

All five routes share the same stations. A line stops where its ceremony applies and passes through where it doesn't. Station color = who owns that stop.

route guide

When to ride which line

FAST — the prose lane

Allowlisted prose-only changes (docs, comments) with positive evidence the diff touches nothing risky. No spec stage at all.

clarifying ⟲ → research ⟲ → implementing → phase_review ⟲ fixing → termination_review → done

Guardrail: hard-risk paths (specs, schemas, skills, engine code, secrets…) force profile_floor: full — you cannot ride FAST past them, and downgrades below the floor route to the human.

STANDARD — the default

Single-implementer development with a locked, prativadi-reviewed plan and per-phase gates. New scaffolds land here.

clarifying ⟲ → research ⟲ → spec ⟲ review → LOCK → per phase: implementing → phase_review ⟲ fixing → termination_review → done

Also carries: the amendment loop back to spec, and the capped review_of_review ⇄ counter_review valve for genuine disputes.

FULL — the adversarial lane

Both agents implement concurrently and audit each other. Mandatory for hard-risk paths; the strongest guarantees in the system.

clarifying ⟲ → …LOCK → per phase: parallel_implementing → test_creation → cross_review ⟲ cross_fixing → deep_review ⟲ phase_fixing → deslop → next phase… → termination_review → done

Team stops are superset-guarded (a retry can't erase the peer's recorded work), deep review gains security/integration angles on risky diffs, and the done gate stacks five independent evidence checks.

RESEARCH — answer, then maybe seed

Question in, reviewed evidence out. The optional spur through spec_drafting turns findings into a locked-plan seed for a follow-on development run.

clarifying ⟲ → research ⟲ review → (optional spec spur) → termination_review ⟲ fixing → done

REVIEW — verdict on existing work

Point it at a diff or artifact. Scope, deep review with a direct fix edge, cleanup, verdict. Nothing new gets built.

clarifying ⟲ → scope ⟲ → deep_review ⟲ phase_fixing → deslop → termination_review → done

mixing lines

phase_profiles: switch trains mid-journey

A full run can declare, at spec time, that a mechanical phase deserves less ceremony: phase_profiles: {"2": "standard"}. The engine pins each phase's entry state to that phase's profile — and per-phase risk floors still bind, so a hard-risk phase can never be demoted.

the amber line

You are a station on every route

human_question

Quick answer needed — legal pre-lock and mid-implementation from any working state. Resume restores the exact prior state, team roster included. wait exit 12

human_decision

You re-route the run: any non-terminal state can escalate here, and it's the only way to reopen done or abandoned. wait exit 11

abandoned

The clean death. From a human pause you can end the run outright — archived, resolver forgets it, commit gate releases. wait exit 13

Delivery is native: the Claude Code-hosted session owns surfacing every pause to you in-session, and the remote session puts that conversation on your phone — no extra plumbing. Meanwhile the other session appends --through-human to its wait: it sleeps through the pause and resumes itself the moment your answer lands. For fleets, the optional dvandva watchdog cron scan logs any baton that stops moving mid-work.

boarding

Get started

# the runtime (one binary, published on crates.io)
cargo install dvandva --version 2.0.0-beta.2

# the plugin, into Claude Code and Codex
dvandva install

Then open two sessions on a feature branch:

Claude:  Implement <feature> with Codex review. Start a Dvandva development run in walkaway mode.
Codex:   Join the same Dvandva development run and review the baton.

Walkaway mode drives the baton autonomously until done — or until a question needs you, in which case the Claude session asks you directly, wherever your remote session is open.
