द्वन्द्व · dvandva — a compound of two equal, opposing members

Dvandva — a governed loop for cross-vendor adversarial execution

One rule generates the whole system: the reviewer is never the author. A model from a different training lineage attacks every step before the loop can call anything done — and a hook makes "done" something you earn with digest-bound evidence, not something you declare.

An agent could promise to self-review, but a promise isn't a gate. Dvandva makes the adversarial pass structural: digest-bound evidence (append-only by convention), cross-family verdicts, and — on Claude Code — a Stop hook that re-blocks the turn until the evidence earns it (bounded: eight consecutive blocks force-end; Codex gates via an invoked CLI/pre-commit checkpoint).

a skill + a hook + goal-designed workflows — no daemon, no binary, no second session propose → stamp → attack → gate, per step evidence-gated done — declaring victory early just blocks three configs: claude+codex · claude-only · codex-only honest enforcement: procedure fail-closed, authenticity stated as advisory

the product

The loop keeps the moat and drops the engine

Dvandva distills adversarial AI review to what the moat actually requires: the reviewer is never the author. One chair session drives every step through propose → stamp → attack → gate; a hook blocks the turn from ending done until each step carries passing cross-reviewer evidence digest-bound to the artifact bytes (a bounded fail-closed nudge on Claude Code; an explicitly invoked CLI/pre-commit checkpoint on Codex). No daemon, no second live session, no binary — a skill, a hook, and a workflow the chair designs per goal (parallel Codex execute lanes, one serialized stamp lane, other-family attack lanes; reviewed building blocks + a runnable flat-case example ship in the plugin), all as the dvandva plugin (repo ↗).

Cross-vendor · strong default

Claude chairs; Codex executes and attacks (gpt-5.6-sol/terra). Evidence must come from a different vendor family than the step's author. Different training, different blind spots.

Claude-only

No Codex subscription needed: fresh Claude subagents across tiers attack (opus attacks sonnet's work). Evidence must carry a reviewer agent id distinct from the author's. Honest degradation — shared-vendor blind spots can survive.

Codex-only

Codex drives; sol attacks terra's work across effort tiers. No Stop hook exists here, so the same predicate gates through gate-cli.sh — invoked explicitly or as a Git pre-commit hook.

Enforcement is stated honestly: the gate fails closed on the procedure — evidence present, cross-reviewer, digest-bound, verdict-passing — and is a bounded nudge (Claude Code force-ends after eight consecutive blocks), aimed at a lazy chair, not a malicious one. It cannot prove a provider call really happened. Two companion skills, delegating-to-codex and delegating-to-grok, teach the chair to dispatch the other models well — self-contained briefs, hardened invocations, independently verified results. Wiring per configuration: adversarial-loop/README.md ↗.

why it exists

Agents grade their own homework. Dvandva takes the pen away.

Decorrelated review

The reviewer is never the author — a different vendor family when two are available (Claude and GPT have systematically different blind spots), a distinct fresh-context agent otherwise. Self-review earns zero credit, structurally.

Proof, not prose

A goal cannot end because the agent says it's done. Every step needs review evidence bound by sha-256 to the exact artifact bytes, from a reviewer that recomputed the digest itself.

Failures stay on record

A fail verdict is never overwritten. Fixes bump the revision, re-stamp, and earn a fresh attack — the gate only ever honors the latest attempt against the current bytes.

proven live

On its first real goal, the loop caught a real bug

Acceptance test, run end-to-end with the native Workflow tool: two Codex lanes built a small CLI tool and its README in parallel, one stamp lane bound the digests, and two independent Opus lanes attacked the results. One artifact passed. The other didn't — the README documented ./wordfreq.sh, but the shipped scripts weren't executable: a cross-step integration bug neither author noticed, caught by the opposing family on the first lap.

The same review pressure built the product itself: the loop's own skills, hook, and workflow blocks went through repeated cross-family attack rounds before shipping, and the reviewers' findings — including a live demonstration that a blocked agent will fabricate evidence rather than dispatch a real review — are why the enforcement section below reads the way it does.

Get started

# Claude Code
claude plugin marketplace add axatbhardwaj/Dvandva
claude plugin install dvandva@dvandva

# Codex
codex plugin marketplace add axatbhardwaj/Dvandva
codex plugin add dvandva@dvandva

No binary, no build step — the plugin (2.0.2) ships the skills, the Stop-hook gate, and the workflow building blocks (the chair designs the workflow per goal; a flat-case example runs as-is). The gate is inert until a session creates a goal; add .adversarial-loop/ to your project's .gitignore.

Then, in one session on a feature branch:

Use the adversarial-loop skill: build <feature>, cross-vendor mode.
Every step gets attacked before it counts — don't end until the gate lets you.

The skill creates .adversarial-loop/goal.json bound to the session, drives propose → stamp → attack → gate per step, and the hook blocks done until passing cross-reviewer evidence exists (bounded on Claude Code; invoked checkpoint on Codex). Claude-only and Codex-only setups use the same loop with fresh-context reviewers — see the config cards above and adversarial-loop/README.md for per-config wiring.

lineage

Distilled from a heavier machine that proved the idea

Dvandva v1–v3 ran the same bet as a two-session protocol: two live agent harnesses passing a validated baton file through a typed state machine, enforced by a Rust binary. Across ~20 real runs it kept proving the moat — cross-vendor review catches bugs neither model catches alone — and kept proving the cost: dead-peer watchdogs, mutual-wait choreography, and a state machine that took more care than the work it governed. The loop keeps the moat and drops the machine. The final engine release is dvandva 3.4.1 on crates.io ↗; its full source, protocol docs, and transition tables live at tag v3.4.1 ↗.
